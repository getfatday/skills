# Engineering Vocabulary (Shared)

Canonical definitions for the engineering domain. Individual avatars extend with expert-specific terms.

| Term | Definition | Use This | Not This |
|------|-----------|---------|----------|
| Red | Failing test. Starting state of every TDD cycle. | "The test is red" | "The test failed" |
| Green | Passing test. Simplest possible implementation. Green is not done. | "Get to green" | "It works" |
| Refactor | Restructure without changing behavior. Only when green. | "Refactor" | "Rewrite" or "clean up" |
| Clean code | Simple, direct, reads like prose, does one thing, no surprises. | "Clean code" | "Working code" |
| Coupling | Cost of change propagation. Changing A forces changes to B. | "Coupling" | "Dependency" |
| Cohesion | Benefit of related things being together. | "Cohesion" | "Organization" |
| Baby steps | Smallest possible increment. Minutes between greens. | "Baby steps" | "Iterations" |
| Technical debt | Cost of expedient solutions that are harder to change later. | "Technical debt" | "Bugs" |
| Boy Scout Rule | Leave code cleaner than you found it, every touch. | "Boy Scout Rule" | "Refactoring sprint" |
| Code smell | Surface indication of a deeper design problem. Signal, not bug. | "Code smell" | "Bug" |
| Professionalism | Responsibility, tests, saying no when necessary. | "Professional obligation" | "Best practice" |
| Sustainable pace | Rate you can maintain indefinitely. Overtime is a symptom. | "Sustainable pace" | "Work-life balance" |
| Composed method | Method at one abstraction level. Does one thing or calls things. | "Composed method" | "Small method" |
| Separation of concerns | Distinct responsibilities in distinct modules. | "Separation of concerns" | "Modularity" |
| Abstraction | Hiding implementation behind stable interface. | "Abstraction" | "Making it generic" |
